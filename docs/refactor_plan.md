# Refactor Plan

Generated by scanning the `job_search_tool` codebase after Prompts 21–24.

---

## 1. Duplicated Logic

### 1.1 `_load_prompt()` duplicated in every chain module
- **Files:** `chains/analyzer.py:20-22`, `chains/tailorer.py:19-21`, `chains/cover_letter.py:18-20`, `chains/followup.py:16-18`
- **Problem:** Identical 3-line helper copied in all four chain files.
- **Fix:** Extract to `chains/utils.py` as `load_prompt(name: str) -> str` and import from there.
- **Complexity:** Low

### 1.2 Resume parse + error handling duplicated in `analyze` and `apply`
- **Files:** `main.py:226-242` (analyze), `main.py:336-350` (apply)
- **Problem:** The same `with console.status(...)` block with identical `FileNotFoundError` and `ValueError` handling appears twice.
- **Fix:** Extract `_parse_resume_for_cli() -> str` in `main.py` that wraps `parse_resume(config.RESUME_PATH)` with the status spinner and error handling.
- **Complexity:** Low

### 1.3 `Progress(SpinnerColumn(), TextColumn(...))` duplicated for every LLM call
- **Files:** `main.py` (analyze, tailor, cover letter, followup, status updates throughout)
- **Problem:** The same 5-line `with Progress(...) as progress: progress.add_task(...)` pattern repeats ~6 times.
- **Fix:** Extract `_run_chain_with_spinner(description: str, fn: Callable[[], T]) -> T` in `main.py` that wraps the Progress context and generic exception handling.
- **Complexity:** Low

### 1.4 Exception handling pattern duplicated for every chain call
- **Files:** `main.py` throughout `analyze`, `apply`, `followup`
- **Problem:** `except ConnectionError: handle_error("Cannot reach Ollama.", ...)` and `except Exception as exc: handle_error(f"Error during X: {exc}", ...)` repeated verbatim.
- **Fix:** Collapse into the `_run_chain_with_spinner` helper above, or create `_handle_chain_error(exc, step_name)`.
- **Complexity:** Low

## 2. Long Functions

### 2.1 `apply()` — ~170 lines
- **File:** `main.py:332-500`
- **Problem:** Parses resume, gets input, validates, analyzes, displays, confirms, checks duplicate, tailors, generates cover letter, saves files, prompts for source/notes, updates tracker.
- **Fix:** Break into:
  - `_gather_inputs()` — parse resume + get job description
  - `_run_analysis(resume, jd)` — analyze + display
  - `_run_tailoring_and_cover_letter(analysis, resume, jd, skip_tailor)` — return tailored + cover letter
  - `_save_outputs(tailored, cover_letter, analysis)` — write files + tracker
- **Complexity:** Medium

### 2.2 `tracker()` — ~60 lines
- **File:** `main.py:618-678`
- **Problem:** Handles show, delete, and edit in one function with many branches.
- **Fix:** Keep as one Typer command (it is a single subcommand with flags), but extract private helpers:
  - `_tracker_delete(company, role)`
  - `_tracker_edit(company, role, field, value)`
- **Complexity:** Low

### 2.3 `_display_analysis()` — ~50 lines
- **File:** `main.py:275-324`
- **Problem:** Not excessively long, but the table-building logic for each section is repetitive.
- **Fix:** Extract `_render_bullet_table(title, items, icon)` helper.
- **Complexity:** Low

## 3. Inconsistent Patterns

### 3.1 `load_workbook` mode inconsistency
- **Files:** `utils/tracker.py:79` (read_only=True), `utils/tracker.py:253` (read_only not set)
- **Problem:** `application_exists` opens read-only, but `update_status`, `delete_application`, `edit_application` open read-write without specifying.
- **Fix:** Always use `read_only=True` for pure reads; explicitly open read-write for mutations. Document why.
- **Complexity:** Low

### 3.2 Error handling style
- **Files:** `main.py` throughout
- **Problem:** Some places call `handle_error()` directly; others do `console.print(...)` + `raise typer.Exit(0/1)`.
- **Fix:** Standardize on `handle_error()` for fatal errors and `console.print(...)` + `raise typer.Exit(0)` for user aborts.
- **Complexity:** Low

### 3.3 Console instantiation
- **Files:** `utils/tracker.py:15` imports `Console` but only `show_tracker` uses it. `utils/resume_parser.py:15` creates `_console` at module level. `config.py:16` imports `Console`.
- **Problem:** Multiple modules create or import `Console` independently.
- **Fix:** Not a critical issue, but consider a shared `utils/console.py` with a singleton console if more modules need it.
- **Complexity:** Low

### 3.4 Path usage
- **Files:** `main.py` uses `config.TRACKER_PATH` (Path) directly, but passes `cl_out` (Path) to `add_application` which expects `str | Path`.
- **Problem:** Inconsistent — fine in practice, but `str(cover_letter_path)` is called inside `add_application` anyway.
- **Fix:** Already handled by explicit `str()` coercion in `add_application`. No action needed.

## 5. Chain Structure

### 5.1 Chain granularity
- **Assessment:** Each chain does exactly one thing (analyze, tailor, cover letter, follow-up). This is correct.
- **No merge needed.**
- **No split needed.**

### 5.2 LangGraph workflow
- **Assessment:** The current pipeline in `main.py` and `app.py` is strictly sequential:
  1. analyze -> 2. tailor -> 3. cover letter -> 4. save/tracker
- **Conclusion:** A LangGraph workflow would be over-engineering. The sequential flow is clear and the dependencies are real (tailor and cover letter both need analyzer output). **Do not adopt LangGraph.**

## 6. Streamlit & CLI Overlap

### 6.1 Resume parsing
- **Files:** `main.py` and `app.py` both call `parse_resume(config.RESUME_PATH)` with similar error handling.
- **Fix:** Extract a shared `_get_resume_text()` helper that returns the resume string or raises a uniform exception. Both CLI and Streamlit can call it.
- **Complexity:** Low

### 6.2 Analysis result display
- **Files:** `main.py:_display_analysis()` and `app.py:analyze mode display`
- **Problem:** Rich tables in CLI vs Streamlit expanders/metrics in UI — these necessarily differ. However, the *data extraction* (which fields to show) is duplicated logic.
- **Fix:** Add a `JobAnalysis.to_display_dict()` method or a small formatter that returns a structured dict of sections. Both CLI and UI can render from it. **Nice-to-have, not critical.**
- **Complexity:** Medium

### 6.3 Full apply pipeline
- **Files:** `main.py:apply()` and `app.py:Full Application mode`
- **Problem:** The orchestration logic (analyze -> tailor -> cover letter -> save -> tracker) is written twice.
- **Fix:** Extract a pure function `run_apply_pipeline(resume, job_description, source, skip_tailor, dry_run)` in a new `pipelines.py` module. It returns a dataclass with `analysis`, `tailored_resume`, `cover_letter`, `saved_paths`. Both CLI and Streamlit call this. The UI gets progress callbacks via a generator or simple step enum.
- **Complexity:** High (requires refactoring both entry points)
