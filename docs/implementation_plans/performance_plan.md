# Performance Plan

Generated by scanning the `job_search_tool` codebase after Prompts 21–28.

---

## 1. Sequential LLM Calls That Could Be Parallelized

### 1.1 Resume tailoring and cover letter generation are independent
- **Dependency graph:**
  ```
  parse_resume -> analyze_job
  analyze_job  -> tailor_resume
  analyze_job  -> generate_cover_letter
  tailor_resume -> save_files
  generate_cover_letter -> save_files
  ```
- **Finding:** `tailor_resume` and `generate_cover_letter` both depend *only* on `analyze_job` output. They do **not** depend on each other.
- **Current behavior:** They run sequentially in both `main.py:apply()` and `app.py`.
- **Fix:** Run them concurrently using `asyncio.gather()` with async LangChain invocations. Both chains are LCEL pipes that can be invoked via `await chain.ainvoke(...)`.
- **Estimated time saved:** ~40–50% of the tailoring + cover letter phase (whichever is slower becomes the only delay).
- **Complexity:** Medium

### 1.2 Analyzer is a hard dependency for everything else
- **Finding:** The analyzer must run first because `tailor_resume` and `generate_cover_letter` need `JobAnalysis`.
- **No parallelization possible here.**

---

## 2. Redundant Work

### 2.1 Resume is re-parsed from PDF on every invocation
- **Finding:** `parse_resume()` opens `pdfplumber`, iterates every page, extracts text, and joins pages. This happens on every `analyze`, `apply`, and Streamlit rerun.
- **Fix:** Cache the parsed resume text to `resume/resume_cache.txt`. Store the PDF's last modification timestamp alongside it. On the next invocation, only re-parse if the PDF mtime has changed.
- **Estimated impact:** Saves 0.5–2 seconds per invocation (depending on PDF size).
- **Complexity:** Low

### 2.2 Job description is re-fetched for the same URL
- **Finding:** If the user runs `apply --url` twice for the same posting, the page is fetched and parsed both times.
- **Fix:** Simple JSON cache at `data/url_cache.json`. Key = URL, value = `{"text": "...", "cached_at": "2026-06-12T10:00:00"}`. Expire after 24 hours.
- **Estimated impact:** Saves the fetch + BeautifulSoup parse time on repeat URLs (~1–3 seconds).
- **Complexity:** Low

---

## 3. LLM Call Optimization

### 3.1 Full resume sent to every chain
- **Finding:** The entire resume text is passed to analyzer, tailorer, and cover letter chains. For long resumes, this inflates token count.
- **Assessment:** The tailorer and cover letter chains genuinely need the full resume to preserve sections and reference specific evidence. The analyzer could theoretically use a shortened version, but it needs to evaluate every skill mention. **Not a quick win.**
- **Fix:** Two-stage approach is over-engineering for this project. **Skip.**

### 3.2 Prompt lengths
- **Finding:** All prompts are already concise. No bloated instructions or unnecessary text. **No action needed.**

---

## 4. Streamlit Performance

### 4.1 Resume parsing already cached
- **Finding:** `app.py` uses `@st.cache_resource` for `_cached_resume_text()`.
- **Status:** Already optimized in Prompt 24. **No action needed.**

### 4.2 Analysis results lost on rerun
- **Finding:** When the user clicks a button or changes a widget, Streamlit reruns the script. Analysis results are not stored in `st.session_state`, so they disappear if the user interacts with another widget.
- **Fix:** Store `analysis`, `tailored_resume`, and `cover_letter` in `st.session_state` and gate re-computation behind session-state checks.
- **Complexity:** Medium
- **Assessment:** This is a UX issue as much as a performance issue. It causes re-analysis on every interaction. **Implement.**

---

## 5. Startup Time

### 5.1 Chain imports at top of `main.py`
- **Finding:** `main.py` imports all chain modules at the top level. For commands like `tracker --show` or `status`, these imports are unnecessary.
- **Fix:** Lazy-import chain modules inside the functions that need them (e.g., `analyze()`, `apply()`, `followup()`). This shaves ~100–300 ms off cold starts for non-LLM commands.
- **Complexity:** Low
- **Assessment:** Minor gain, but trivial to implement. **Implement.**

### 5.2 `config.py` creates directories on import
- **Finding:** `config.py` runs `mkdir(parents=True, exist_ok=True)` for output directories at module import time.
- **Assessment:** This is cheap and harmless. **No action needed.**

## Prioritized Improvements

| Priority | Finding | File | Impact | Complexity |
|----------|---------|------|--------|------------|
| 1 | Resume text cache by PDF mtime | `utils/resume_parser.py` | Medium | Low |
| 2 | URL cache with 24h expiry | `utils/helpers.py` or `main.py` | Medium | Low |
| 3 | Async parallel tailoring + cover letter | `main.py`, `app.py` | High | Medium |
| 4 | Use `st.session_state` for results | `app.py` | Medium | Medium |
| 5 | Lazy-import chain modules in CLI | `main.py` | Low | Low |
